1
Browser sends multipart POST request

User selects an image and clicks Translate. The browser uses FormData to pack the image file + target language into a multipart/form-data HTTP POST request to /api/translate.

POST /api/translate
Content-Type: multipart/form-data

Parts:
  file      → binary image bytes (PNG/JPEG)
  targetLang → "Hindi"
💡 Multipart means the HTTP body is split into "parts" — one for the file bytes, one for each text field. Spring's @RequestPart and @RequestParam unpack these automatically.
2
TranslateCtrl receives and validates

TranslateCtrl.java is the entry point. Spring Boot routes the HTTP request here because of @PostMapping("/translate"). It does 3 checks:

1. file.isEmpty()          → reject if no file sent
2. content-type check      → only image/png, image/jpeg, image/webp allowed
3. file.getBytes()         → reads all image bytes into memory as byte[]

The byte[] (raw image bytes) is then passed to VisionService. At this point the image is just bytes — not yet decoded into pixels.

3
VisionService — sends image to GPT-4o

This is the core AI step. The image bytes + a text prompt are sent to OpenAI's GPT-4o model via Spring AI's ChatClient.

chatClient.prompt()
  .user(u -> u
      .text(prompt)              ← text instruction
      .media(mimeType, bytes)    ← image bytes
  )
  .call()
  .content()                     ← GPT-4o's text response

GPT-4o reads the image visually (like a human would) and returns a JSON array listing all text it found, the translation, and bounding box coordinates.

💡 This single API call does what used to require TWO separate systems: Tesseract (OCR) + Google Translate. GPT-4o handles both in one shot.
4
JSON parsing → TextRegion objects

GPT-4o returns a raw JSON string. Jackson's ObjectMapper parses it into a List<TextRegion>:

// GPT-4o returns this string:
[
  {
    "original_text": "Translate this text",
    "translated_text": "इस पाठ का अनुवाद करें",
    "x_percent": 17.58,   ← left edge = 17.58% of image width
    "y_percent": 39.06,   ← top edge  = 39.06% of image height
    "width_percent": 64.84,
    "height_percent": 20.31
  }
]

// Jackson maps this to:
TextRegion {
  originalText  = "Translate this text"
  translatedText= "इस पाठ का अनुवाद करें"
  xPct = 17.58, yPct = 39.06
  wPct = 64.84, hPct = 20.31
}

Using percentages (not pixels) means the coordinates work for any image resolution — a 600x200px and a 3000x1000px image both use the same 0-100 scale.

5
EmbedService — drawing onto the image

ImageIO.read() decodes the raw bytes into a BufferedImage — a pixel grid in memory. Then Java2D (Java's built-in graphics library) draws onto it:

Step A: Convert % coords → pixels
  x = (17.58 / 100) × 600 = 105 px
  y = (39.06 / 100) × 200 =  78 px
  w = (64.84 / 100) × 600 = 389 px
  h = (20.31 / 100) × 200 =  41 px

Step B: Sample background color
  Read pixels in that area → average RGB → use as fill color

Step C: Erase original text
  g.fillRect(x, y, w, h)  ← paint over original text

Step D: Find Hindi-capable font
  Try "Noto Sans Devanagari" → test canDisplayUpTo() → use it

Step E: Auto-fit font size
  Start at h × 0.72 = ~30px
  Shrink until longest word fits in width w

Step F: Draw translated text
  g.drawString("इस पाठ का अनुवाद करें", x, y + ascent)
6
Save + Return response

The modified BufferedImage is encoded back to bytes via ImageIO.write(), saved to disk, and returned as HTTP response with Content-Disposition: attachment which tells the browser to auto-download it.

ImageIO.write(canvas, "png", outputStream)  ← pixels → bytes
Files.write(outputPath, resultBytes)         ← save to disk
return new ResponseEntity(bytes, headers, 200) ← send to browser
🧠 What is a Token?

OpenAI doesn't process text word by word — it uses tokens. A token is roughly ¾ of a word or 4 characters in English.

"Hello"         → 1 token
"Translate"     → 1 token
"this text"     → 2 tokens
"इस पाठ का"    → ~6-8 tokens  (non-Latin scripts use more tokens per word)

Rule of thumb:
  100 tokens ≈ 75 English words
  100 tokens ≈ 40-50 Hindi/Arabic words
💡 Everything costs tokens — your prompt, the image, AND the response. You're billed for total tokens in + tokens out.
📊 Token Breakdown for One Image Request

~500-1500

~$0.001-0.005
~100-500

~$0.001-0.005

Total per image: ~$0.003 - $0.01 depending on image size and amount of text.

// Image token calculation (GPT-4o):
Small image  (≤512×512px):  85 tokens  → ~$0.00025
Medium image (600×200px):   ~500 tokens → ~$0.0015
Large image  (2048×2048px): ~1700 tokens → ~$0.005

// Formula for larger images:
tiles = ceil(width/512) × ceil(height/512)
tokens = 85 + (170 × tiles)
💬 How ChatClient Works Internally

Spring AI's ChatClient is a wrapper around OpenAI's REST API. Here's what happens when you call .call().content():

1. ChatClient builds an HTTP request:
   POST https://api.openai.com/v1/chat/completions
   Authorization: Bearer sk-...
   Content-Type: application/json

   Body: {
     "model": "gpt-4o",
     "max_tokens": 4096,
     "messages": [
       {
         "role": "user",
         "content": [
           {
             "type": "text",
             "text": "Detect ALL text in this image..."
           },
           {
             "type": "image_url",
             "image_url": {
               "url": "data:image/png;base64,iVBORw0KGgo..."
                                              ↑ your image as base64
             }
           }
         ]
       }
     ]
   }

2. OpenAI processes it with GPT-4o Vision model
   → model "sees" the image + reads the text prompt
   → generates JSON response

3. OpenAI returns:
   {
     "choices": [{
       "message": {
         "role": "assistant",
         "content": "[{\"original_text\":\"...\"}]"
       }
     }],
     "usage": {
       "prompt_tokens": 823,
       "completion_tokens": 124,
       "total_tokens": 947
     }
   }

4. Spring AI extracts choices[0].message.content
   → returns it as a plain String to your VisionService
💡 The image is converted to Base64 before sending. A 600×200 PNG (~6KB file) becomes ~8KB of Base64 text embedded in the JSON body.
🎯 The Prompt — Why It Matters

The prompt is the most important part. Here's what each line does:

"Detect ALL text in this image."
  → tells GPT-4o to do OCR — find every piece of text

"Translate each text block to Hindi."
  → tells it to translate — combined OCR + translate

"Return ONLY a raw JSON array, no markdown, no explanation."
  → critical! without this, GPT wraps output in ```json fences
  → our cleanJson() strips fences but it's better to prevent them

"Each element must have exactly: {original_text, translated_text,
  x_percent, y_percent, width_percent, height_percent}"
  → forces a strict schema — GPT must return these exact field names
  → Jackson needs exact field names to map to TextRegion class

"If no text found return []."
  → handles empty images cleanly — no null pointer exceptions
⚠️ If GPT-4o ignores your schema and returns different field names (e.g. "text" instead of "original_text"), Jackson will parse it but all fields will be null — resulting in a blank image. This is why @JsonIgnoreProperties(ignoreUnknown=true) is on TextRegion.
💰 Token Optimization Tips
1. RESIZE images before sending
   → 600×400 uses far fewer tokens than 3000×2000
   → Add this before calling visionService:
   
   BufferedImage resized = new BufferedImage(800, 600, TYPE_INT_RGB);
   Graphics2D g = resized.createGraphics();
   g.drawImage(original, 0, 0, 800, 600, null);

2. COMPRESS PNG to JPEG for photos
   → JPEG at quality 80 = ~10x smaller file = fewer image tokens

3. LIMIT max_tokens in response
   → Set spring.ai.openai.chat.options.max-tokens=1000
   → JSON responses rarely need more than 500 tokens

4. CACHE results
   → Same image = same translation = no need to call API again
   → Hash the image bytes, store result in a Map

5. MONITOR usage
   → Check platform.openai.com/usage daily
   → Set hard limit at platform.openai.com/settings/limits
📄 TextRegion.java — The Data Model

This is a simple POJO (Plain Old Java Object) that represents one block of text found in the image. Jackson uses @JsonProperty to map GPT-4o's JSON field names to Java field names.

@JsonProperty("x_percent")   ← GPT returns "x_percent"
private double xPct;          ← Java uses xPct (camelCase)

// Why percentages?
// GPT-4o doesn't know your image size in pixels
// But it can estimate "this text starts 20% from the left"
// EmbedService converts: pixels = (percent/100) × imageDimension
✅ @JsonIgnoreProperties(ignoreUnknown=true) means if GPT adds extra fields (like it added "annotations" to OpenAI responses), Jackson silently ignores them instead of crashing.
📄 AppConfig.java — Spring Bean Wiring

This tells Spring how to create the ChatClient bean — the object that talks to OpenAI.

@Bean
public ChatClient chatClient(ChatClient.Builder builder) {
    return builder.build();
}
// Spring AI auto-creates ChatClient.Builder using your
// application.properties (api-key, model, etc.)
// You just call .build() to get a ready-to-use client

// Why @Bean?
// Makes ChatClient available for @Autowired injection
// everywhere in the app — VisionService gets it via
// @RequiredArgsConstructor (Lombok generates constructor injection)
📄 OpenAiApiConfig.java — Custom RestClient Fix

This exists only to fix the "annotations" unknown field bug. Spring AI M6 uses an internal RestClient with a strict ObjectMapper. We replace it with a tolerant one.

// Problem: OpenAI added "annotations" field to responses
// Spring AI M6's ObjectMapper doesn't know this field
// Result: crash with "Unrecognized field annotations"

// Fix: Create our own ObjectMapper with:
ObjectMapper tolerantMapper = new ObjectMapper()
    .configure(FAIL_ON_UNKNOWN_PROPERTIES, false);
    //          ↑ silently ignore any unknown fields

// Then inject it into OpenAiApi's RestClient
// so ALL responses from OpenAI use this tolerant mapper
📄 VisionService.java — The AI Bridge

Three responsibilities: build prompt, call GPT-4o, parse response.

// 1. Build prompt (targetLanguage from application.properties)
String prompt = "Detect ALL text... translate to " + targetLanguage;

// 2. Wrap image bytes as Spring AI Media object
Media media = new Media(MimeTypeUtils.parseMimeType(mimeType),
                        new ByteArrayResource(imageBytes));
// ByteArrayResource wraps byte[] so Spring AI can read it
// MimeTypeUtils.parseMimeType("image/png") → MimeType object

// 3. Call GPT-4o
String raw = chatClient.prompt()
    .user(u -> u.text(prompt).media(mimeType, bytes))
    .call()
    .content();    // blocks until GPT responds (~2-10 seconds)

// 4. Clean JSON (GPT sometimes wraps in ```json fences)
String cleaned = raw.replaceAll("^```[a-zA-Z]*\n?", "")
                    .replaceAll("```\s*$", "");

// 5. Parse into List<TextRegion>
return objectMapper.readValue(cleaned,
    new TypeReference<List<TextRegion>>() {});
📄 EmbedService.java — Image Manipulation

Pure Java2D graphics. No external library needed — Java has built-in image drawing.

// Key Java2D concepts:
BufferedImage  → pixel grid in memory (like a 2D int array)
Graphics2D     → drawing context (pen) for that image
FontMetrics    → measures text width/height for a given font+size

// deepCopy() — why?
// Never draw on the original image
// If something goes wrong, original is intact
BufferedImage canvas = deepCopy(original);

// sampleColor() — why?
// We need to erase the original text cleanly
// If background is white → fill white
// If background is dark → fill dark
// Samples a grid of pixels and averages their RGB

// contrastColor() — luminance formula
// lum = 0.299×R + 0.587×G + 0.114×B  (human eye weights)
// lum > 0.5 → bright background → use BLACK text
// lum ≤ 0.5 → dark background → use WHITE text

// fitFont() — why auto-fit?
// Translated text may be longer than original
// "Hello" (5 chars) → "नमस्ते" (6 chars + wider glyphs)
// Start at h×0.72, shrink 0.5pt at a time until it fits

⚡ Common Code Mistakes to Watch For
MISTAKE 1: TextRegion field names don't match GPT JSON
  GPT returns:   "x_percent"
  @JsonProperty: "xPercent"   ← WRONG, will be null
  Fix:           "x_percent"  ← must match exactly

MISTAKE 2: Forgetting cleanJson()
  GPT sometimes returns:  ```json\n[{...}]\n```
  ObjectMapper.readValue() crashes on the ``` prefix
  Fix: always strip fences before parsing

MISTAKE 3: Drawing on original instead of copy
  g = original.createGraphics()  ← modifies original!
  Fix: always deepCopy() first

MISTAKE 4: Not handling empty regions
  if (regions.isEmpty()) return original  ← skip drawing step
  Without this: NullPointerException in EmbedService loop

MISTAKE 5: Wrong Content-Disposition
  "inline"     → browser shows image in tab, no download
  "attachment" → browser downloads file automatically
🐳 Dockerfile — Layer by Layer
FROM maven:3.9.6-eclipse-temurin-17 AS builder
  → Use Maven + JDK 17 image for building
  → "AS builder" names this stage — used in COPY --from=builder later

COPY pom.xml .
  → Copy ONLY pom.xml first (not src/)
  → Docker caches this layer
  → If pom.xml unchanged next build → skip dep download!

RUN --mount=type=cache,target=/root/.m2 \
    mvn dependency:resolve -B -q
  → --mount=type=cache persists .m2 folder between docker builds
  → First build: downloads all deps (~2-5 min)
  → Next builds: uses cache (~10 seconds)

COPY src ./src
  → Copy source code AFTER deps cached
  → Source changes don't invalidate dep cache layer

RUN --mount=type=cache,target=/root/.m2 \
    mvn clean package -DskipTests -B -q
  → Build the JAR, skip tests, use cache

FROM eclipse-temurin:17-jre-jammy
  → Fresh minimal image — just JRE, no Maven/JDK
  → "jammy" = Ubuntu 22.04 LTS base
  → Result: smaller final image (~200MB vs ~500MB with JDK)

COPY --from=builder /app/target/*.jar app.jar
  → Only copy the JAR from builder stage
  → Source code, Maven cache NOT in final image ✅

RUN apt-get update && apt-get install -y \

🌍 Environment Variables — How Spring reads them
// In application.properties:
spring.ai.openai.api-key=sk-xxx

// In Docker run command:
-e SPRING_AI_OPENAI_API_KEY=sk-xxx

// Spring Boot auto-converts:
// SPRING_AI_OPENAI_API_KEY (env var, uppercase+underscores)
//    → spring.ai.openai.api-key (properties, dots+hyphens)

// Same for custom properties:
// app.target-language    → APP_TARGET_LANGUAGE
// app.output.directory   → APP_OUTPUT_DIRECTORY

// Priority (highest to lowest):
// 1. Environment variables  ← Docker -e flag wins
// 2. application.properties ← fallback
// 3. @Value default value   ← last resort
✅ Never hardcode API keys in application.properties when using Docker. Always pass via -e environment variable so the key is never baked into the image.
